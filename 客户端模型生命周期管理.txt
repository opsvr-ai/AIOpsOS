## AIOpsOS 客户端：模型与配置的全生命周期管理

为实现对数千异构客户端的集中管控与弹性智能，客户端遵循 **“极简初始、集中下发、本地缓存、动态更新”** 的管理原则。安装后的客户端只保留**与服务端通信的最基本配置**，所有运行态配置、Skills 清单、采集规则、模型参数甚至推理模型文件均由服务端统一管理与下发，客户端在本地安全缓存，即使短暂离线也可正常运行。

---

### 一、客户端初始化：零配置部署

#### 1. 初始安装包内容
- 客户端二进制文件（或 OCI 镜像）。
- 基础配置文件 `agent.yaml`，仅包含：
  ```yaml
  server:
    endpoint: "wss://aiopsos-control.example.com:443"
    auth_type: mtls
    cert_path: /etc/aiopsos/agent/client.crt
    key_path: /etc/aiopsos/agent/client.key
    ca_path: /etc/aiopsos/agent/ca.crt
  ```
- 预置的客户端证书和 CA 证书（由部署工具或人工预埋）。
- 其余所有配置项均为空或默认值，等待服务端下发。

#### 2. 首次启动与注册
- 使用 mTLS 建立 WebSocket 长连接，向服务端 Control Plane 注册。
- 上报本地硬件信息、操作系统、已安装的采集器、沙箱引擎版本等。
- 服务端根据证书指纹识别节点，将其与预定义的 **Agent Profile** 关联，或自动创建默认 Profile。
- 注册成功后，服务端通过 WS 推送**完整有效配置**，并下发 Agent 的当前任务列表。

---

### 二、服务端配置模型：Agent Profile

Agent Profile 是服务端为每一台客户端定义的全量运行配置，存储于 PostgreSQL `agent_profiles` 表，通过管控 API 管理。

#### 1. Profile 结构
```json
{
  "agent_id": "agent-01",
  "profile_version": 12,
  "skills": {
    "allowed": ["restart_service", "collect_logs", "check_health"],
    "denied": ["modify_kernel"]
  },
  "collection": {
    "metrics_interval": 30,
    "log_targets": ["/var/log/syslog", "journald"],
    "filters": ["error", "critical"]
  },
  "rules": {
    "autonomous_actions": [
      {
        "condition": "disk_usage > 90%",
        "action": "cleanup_logs",
        "require_approval": false
      }
    ],
    "max_autonomous_actions_per_hour": 5
  },
  "model_config": {
    "enable_local_llm": true,
    "model_name": "qwen2:7b-instruct",
    "temperature": 0.1,
    "max_tokens": 1024,
    "resource_limit": {
      "cpu_shares": 2,
      "memory_mb": 4096
    }
  },
  "resources": {
    "max_cpu_percent": 50,
    "max_memory_mb": 512,
    "disk_quota_mb": 2048
  },
  "update_policy": {
    "model_sync_mode": "background",
    "config_sync_interval_sec": 60,
    "hot_reload": true
  }
}
```

#### 2. 配置项说明
- **skills**：动态白名单黑名单，控制客户端可执行的操作范围。
- **collection**：采集策略，包括指标推送间隔、日志监听路径、正则过滤。
- **rules**：断网自治规则和频率限制，防止客户端过度操作。
- **model_config**：本地模型配置，决定是否启用本地推理及资源分配。
- **resources**：客户端整体资源限制，防止过度消耗主机资源。
- **update_policy**：配置和模型的同步策略。

#### 3. 管控操作
- 管理员通过前端 **Agent 管理** 页面修改 Profile，提交后版本号自增。
- 支持批量修改（如为所有“数据库服务器”分组统一调整采集间隔）。
- 修改后，Control Plane 通过 WebSocket 推送新配置到对应的 Agent，或 Agent 定期对比版本号主动拉取。

### 三、客户端配置缓存与动态更新

#### 1. 本地缓存
- 客户端将服务端最后一次下发的完整配置以加密文件形式存储在 `/opt/aiopsos/agent/cache/config.enc`。
- 配置包含数字签名，客户端在加载时使用服务端公钥验签，防止篡改。
- 启动时若无法连接服务端，则加载缓存配置继续运行；连接恢复后自动同步最新版本。

#### 2. 热更新机制
- 客户端监听 WebSocket 的 `config_update` 消息，收到后解析新配置并执行差分应用：
  - 更新 Skills 白名单，重新加载沙箱权限。
  - 调整采集指标间隔，无需重启。
  - 启动或停止本地 LLM 进程（若 `model_config.enable_local_llm` 变化）。
  - 更新自治规则引擎。
- 所有更改记录日志并上报服务端，做到可追溯。

### 四、模型下发策略

对于启用本地推理的客户端，模型文件由服务端统一分发，客户端无需手动下载。

#### 1. 模型仓库
- 服务端维护一个 **模型注册表（Model Registry）**，存储常用 GGUF 格式模型。
- 模型文件可存储于 S3、MinIO 或共享文件系统，按版本管理。

#### 2. 按需下发
- 当 Agent Profile 中指定了 `model_name` 且客户端尚未缓存时，Agent 向服务端请求模型文件元数据（大小、SHA256、分片信息）。
- 支持**分片断点续传**，适应大文件及不稳定网络。
- 模型文件下载后校验哈希，解压至 `/opt/aiopsos/agent/models/`，并通知 Ollama 等推理引擎加载。

#### 3. 更新与清理
- 当 Profile 改变模型名称或版本时，客户端停止旧模型线程，下载新模型，切换后删除旧文件。
- 若 `enable_local_llm` 变更为 false，卸载模型并释放资源。

#### 4. 安全保护
- 模型包由服务端私钥签名，客户端验证。
- 传输通道复用 mTLS 加密，确保模型不被篡改或窃取。

---

### 五、与管控分离架构的融合

- **Control Plane** 管理所有 Agent Profile 和模型上传，提供 API 与前端。
- **Execution Plane** 仅负责将任务下发给客户端以及接收结果，不关心配置管理。
- **Agent 通信**：WebSocket 连接指向 Control Plane 的特定端点（`/agent-gateway`），Control Plane 进行配置同步和心跳管理；执行指令可由 Execution Plane 通过 Control Plane 代理转发，或 Agent 直接同时维持与 Execution Plane 的轻量连接（用于实时任务），架构灵活。

这种集中管控机制使运维团队能够像管理代码一样管理数千客户端智能体，实现一致的运维标准、快速的策略调整，且完全满足企业安全与合规要求。
