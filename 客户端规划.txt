## AIOpsOS 客户端规划：从单一平台到云边端智能体矩阵

要将平台能力延伸至被管对象，**AIOpsOS Client Agent（后称 Agent）** 是安装在服务器、虚拟机、容器、网络设备或存储上的轻量级智能代理。它与平台服务端通过安全通道协同，接收指令、执行 Skills、采集数据、进行本地自治，从而真正形成“感知-决策-执行”闭环。

### 一、客户端定位与核心价值

- **定位**：平台在边缘侧的智能执行器，是主智能体手脚的延伸，实现 **1分钟发现（已有） → 5分钟定位（服务端+客户端协同） → 10分钟恢复（客户端执行）** 的最后一步。
- **核心价值**：
  - **安全的自动化操作**：通过客户端下发的 Skills，实现安全的软件部署、服务重启、配置变更等操作。
  - **实时数据采集与本地预处理**：无需穿透网络，就地采集系统指标、日志、进程信息，并进行降噪、压缩后上报。
  - **断网自治**：网络中断时，仍可根据预设规则执行健康检查、自动修复，待网络恢复后同步结果。
  - **异构设备统一管理**：屏蔽服务器、网络设备、存储系统的协议差异，对外提供一致的操作接口。

### 二、客户端整体架构

#### 1. 组件模型
```
┌─────────────────────────────────────────┐
│         AIOpsOS Control Plane           │
│  - 客户端注册/审批                     │
│  - 下发 Skills & 配置                  │
│  - 权限分配 (RBAC for Agents)          │
└───────────────┬─────────────────────────┘
                │ (mTLS + gRPC/WebSocket)
┌───────────────▼─────────────────────────┐
│          AIOpsOS Client Agent           │
│  ┌─────────────────────────────────────┐│
│  │ 通讯与服务发现                      ││
│  └───────────────┬─────────────────────┘│
│  ┌───────────────▼─────────────────────┐│
│  │ 安全执行引擎 (沙箱)                 ││
│  │  - Docker/WebAssembly?             ││
│  │  - 资源限制/系统调用过滤           ││
│  └───────────────┬─────────────────────┘│
│  ┌───────────────▼─────────────────────┐│
│  │ 本地智能体（可选）                  ││
│  │  - 规则引擎                        ││
│  │  - 小型 LLM (Ollama)               ││
│  └───────────────┬─────────────────────┘│
│  ┌───────────────▼─────────────────────┐│
│  │ 平台能力                            ││
│  │  - 系统采集器 (metrics/logs/proc)  ││
│  │  - Skill 加载与执行 (HTTP/脚本)    ││
│  │  - 生命周期管理 (自更新)            ││
│  └─────────────────────────────────────┘│
└─────────────────────────────────────────┘
```

#### 2. 运行时形态
- **物理机/虚拟机**：以守护进程形式安装，支持 Linux (systemd)、Windows Service。
- **容器化环境**：以 Sidecar 容器或 DaemonSet 部署。
- **网络设备/存储**：若设备不支持直接安装，可通过**代理网关模式**，由相邻服务器上的 Agent 使用 SSH/netconf/IPMI 等协议进行代理管理。但在平台视角，它仍然是一个“客户端实体”。

### 三、核心功能模块

#### 1. 安全通讯与注册
- **双向 TLS 认证**：Agent 启动时携带签发证书，向服务端注册，服务端根据证书指纹识别 Agent 身份。
- **注册信息**：主机名、IP、操作系统、硬件资源、已安装能力（采集器、沙箱、可对接协议等）。
- **心跳与健康监控**：定期上报心跳，服务端标记在线状态，异常离线触发告警。

#### 2. Skill 管理
- **远程下发与更新**：服务端通过管控API将已注册的 Skill（如 restart_service, collect_diagnosis）下发到指定 Agent，Agent 从 Skill Registry 下载代码包并加载至沙箱。
- **本地 Skills 目录**：`/opt/aiopsos/agent/skills/`，每个 Skill 有元数据描述权限要求、资源限制等。
- **执行请求**：由服务端主智能体规划后，调用 Skill 执行，参数通过 RPC 传递，执行结果返回。
- **风险控制**：所有下发的 Skill 必须经过服务端沙箱扫描和审批，Agent 本地执行时也采用相同沙箱策略（Docker/Wasm）。

#### 3. 安全执行引擎
- **沙箱环境**：复用服务端的 Skill 沙箱设计，使用 gVisor 或 Firecracker 进行隔离，限制网络白名单、文件系统只读、禁止容器逃逸。
- **操作权限**：依据 Role-Based Agent Control，不同 Agent 被授权执行不同类别的 Skills（如只读采集 vs. 服务启停）。
- **审计日志**：所有操作记录回传服务端，不可篡改。

#### 4. 本地数据采集与预处理
- **系统指标**：CPU、内存、磁盘、网络、进程等；复用 Prometheus Node Exporter 或自研轻量采集。
- **日志采集**：支持 journald、文件尾部跟踪，经正则解析和压缩后流式上传至 Kafka 或直接推服务端。
- **事件触发**：当本地指标超过阈值（可动态配置），可生成事件并通知服务端，触发告警分析流程。
- **智能降噪**：本地进行初步去重和聚合，减少网络负载。

#### 5. 本地自治能力
- **规则引擎**：服务端可下发简单的触发规则和恢复动作，即使断网，Agent 也可自主执行 Health Check 和 Restart。
- **本地轻量 LLM（可选）**：在资源允许的机器上，部署 Ollama 加载 7B 模型，用于日志分析、异常检测，决策路径短时延更低，并与云端主智能体协同。
- **边缘推理结果**：本地分析结论可实时上报，也可在连接恢复后批量同步。

#### 6. 多设备类型适配
- **通用服务器 Agent**：Linux/Windows，支持 Docker 沙箱，全面采集与执行。
- **虚拟化平台 Agent**：通过 vCenter/Proxmox API 管理虚拟机生命周期、快照、迁移等，其本身是一个运行在管理节点上的 Agent，但操作目标是虚拟化平台。
- **存储系统 Agent**：对接存储管理接口（如 NetApp ONTAP API、Ceph RBD），执行卷管理、快照等操作。
- **网络设备 Agent**：通过 NETCONF/RESTCONF 或 SSH 代理，管理交换机、路由器。这些 Agent 可能部署在一个网络管理服务器上，负责与网络设备通信，平台将其视为一个“网络设备代理 Agent”。

### 四、与服务端的协同交互

#### 1. 任务派发模式
- **主动拉取**：Agent 与服务端保持 WebSocket 长连接，监听指令，适合自然通信的服务器 Agent。
- **服务端推送**：管控平面通过 gRPC 直接触发 Agent 执行，适合需立即响应的场景。
- **消息队列**：使用 Kafka 解耦，服务端发布任务到 Agent Topic，Agent 消费执行，适合异步操作。

#### 2. 数据流
- **采集数据上报**：Metrics 可走远程写入 VictoriaMetrics；Logs 送 Kafka；事件可走 WebSocket 或 REST 批量上报。
- **执行结果回传**：同步或异步返回结果，包含退出码、标准输出、错误信息。

#### 3. 权限管控同步
- 服务端 RBAC 体系扩展 **Agent Identity** 权限，每个 Agent 在注册时分配角色（如 “Linux生产服务器”，“网络核心设备”）。
- 角色决定可执行的 Skill 集合，当服务端下发 Skill 或执行请求时，自动校验权限。

### 五、安全与合规

- **证书生命周期**：Agent 证书自动轮换，由服务端 CA 签发。
- **最小权限原则**：Agent 运行用户非 root，仅授权必要系统调用。
- **代码签名**：下发的 Skill 包由服务端私钥签名，Agent 验证签名后加载。
- **全链路审计**：从平台操作者 → 主智能体 → Agent → 最终设备动作，完整记录，满足合规。

### 六、部署与管理

- **安装方式**：提供一键安装脚本，支持批量部署（Ansible Playbook）。
- **自更新**：Agent 自身支持热更新，接收服务端下发的升级指令，拉取新版本并重启。
- **监控自身**：Agent 健康状态、资源消耗、心跳等均可在服务端仪表盘查看。

### 七、与现有 AIOpsOS 模块的配合

- **智能体操作界面**：用户可通过对话直接说“帮我重启 10.1.2.3 上的 pay 服务”，主智能体会规划出调用 `restart_service` Skill，并由 `execution plane` 下发到对应 Agent。
- **告警中心**：Agent 本地检测到异常（如磁盘满），生成告警事件并上报，自动触发根因分析，分析后可能生成修复建议，确认后可下发修复动作到 Agent。
- **知识库**：通过 Agent 执行的修复过程，可沉淀为知识条目（当告警确认并入知识库时序列化相关操作）。

### 八、后续迭代路径

- **离线推理扩展**：结合本地模型，实现边缘侧根因初筛，耗时从秒级降到毫秒级。
- **设备拓扑自动发现**：Agent 可通过 LLDP/CDP 等协议自动发现邻居，构建 CMDB 拓扑。
- **跨 Agent 协同**：复杂操作需多个 Agent 配合（如虚拟机迁移涉及计算、存储、网络），由主智能体编排成工作流。

---

通过引入 AIOpsOS Client Agent，平台真正从“观察+建议”升级为“观察+决策+行动”的智能运维操作系统，并能够覆盖服务器、网络、存储全栈，实现 **10分钟恢复** 承诺的关键执行能力。客户端设计延续了平台的插件化、安全沙箱、权限控制等核心理念，确保整套体系的安全与可控。
