## 告警中心设计：从事件到知识闭环

告警中心作为运维中心的核心模块，承接事件接入、自动/手动分析、人工确认及知识沉淀的全流程。它不仅是告警列表，更是连接实时监控与智能分析的枢纽。

---

### 一、模块定位与导航

- **位置**：运维中心 → **告警中心**（与对话、场景运维、自动化等并列）
- **导航图标**：🔔
- **路由**：`/ops/alerts`

---

### 二、告警数据模型

在原有 `ops_events` 基础上抽象出告警实体，支持生命周期管理。

```sql
CREATE TABLE alerts (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_id UUID REFERENCES ops_events(id),
    title VARCHAR(512) NOT NULL,
    source VARCHAR(64) NOT NULL,           -- prometheus, zabbix, custom
    severity VARCHAR(16) NOT NULL,         -- critical, warning, info
    status VARCHAR(32) NOT NULL DEFAULT 'pending', 
    -- pending / analyzing / awaiting_review / confirmed / dismissed / closed
    raw_event JSONB,                       -- 原始事件数据
    enriched_context JSONB,                -- 预取的主机/服务信息
    analysis_result JSONB,                 -- 智能体分析后的结构化结果
    confirmed_by VARCHAR(128),             -- 确认人
    confirmed_at TIMESTAMPTZ,
    knowledge_entry_id UUID,               -- 关联知识库条目
    assigned_to VARCHAR(128),              -- 指派给
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now()
);

-- 索引
CREATE INDEX idx_alerts_status ON alerts(status);
CREATE INDEX idx_alerts_severity ON alerts(severity);
CREATE INDEX idx_alerts_created ON alerts(created_at DESC);
```

### 三、告警生命周期与状态机

```
事件进入 → 生成告警 (pending)
   ↓
   ├─ 自动触发规则匹配 → 自动分析 (analyzing)
   │      ↓
   │   分析完成 (awaiting_review) ← 人工也可在此阶段手动启动分析
   │      ↓
   │   ┌─ 人工确认 (confirmed)
   │   │      ↓
   │   │   (可选) 自动生成知识条目 → 关联 knowledge_entry_id
   │   │      ↓
   │   │   关闭 (closed)
   │   │
   │   └─ 人工忽略/驳回 (dismissed)
   │         ↓
   │      留存，不进入知识库
   │
   └─ 无触发规则 → 待确认 (awaiting_review) 直接由人工判断
```

关键状态说明：
- **pending**：新告警，等待自动触发或人工处理。
- **analyzing**：智能体正在分析中。
- **awaiting_review**：分析完成，等待人工确认。
- **confirmed**：已确认，可沉淀知识。
- **dismissed**：误报或无需处理，忽略。
- **closed**：生命周期结束。

### 四、界面设计（极简科技风）

#### 1. 告警列表页

- **顶部工具栏**：
  - 左侧：告警总数、各状态数量标签（可点击筛选）
  - 右侧：搜索框（标题/主机/服务）、刷新、时间范围选择器
- **列表卡片**（每条告警一个轻量卡片）：
  - 左侧状态指示灯：🔴 critical / 🟡 warning / 🔵 info
  - 中部：标题、来源、主机/服务、触发时间、状态标签
  - 右侧：操作按钮（仅显示最常用的一个主操作）
    - `pending`：▶ 立即分析
    - `awaiting_review`：✅ 确认 / ❌ 驳回
    - `confirmed/dismissed`：详情
  - 点击卡片展开内联详情（不跳转新页），展示原始事件、分析结果、操作历史。
- **批量操作**：支持勾选多条，统一确认/忽略。

#### 2. 告警详情（内联展开）

- 事件摘要：主机、指标、阈值、当前值、时间线。
- 预取上下文（如果已分析，展示智能体推理摘要；若未分析，显示“尚未分析”）。
- 分析结果卡片（如有）：根因推断、建议、关联告警。
- 操作按钮：确认、忽略、重新分析、指派给他人、在对话中打开（跳转到对话页并带入该告警上下文）。
- 知识入库预览：当用户点击确认时，可弹出小表单，允许修改标题、补充标签，预览即将存入的知识片段，点击“确认并入库”完成。

#### 3. 与对话中心联动

- 从告警卡片点击“在对话中打开”，跳转到对话页面，并自动发送一条包含告警详情的消息，启动告警分析场景。
- 对话完成后，分析结果可回填到告警的 `analysis_result` 字段，状态流转至 `awaiting_review`。

---

### 五、自动分析与触发规则集成

告警中心与**自动化 → 触发规则**紧密配合：

- 触发规则中的动作可配置为“**自动分析本告警**”，当新告警符合条件时，自动调用智能体分析，并将结果写入告警，状态变为 `awaiting_review`。
- 若告警没有匹配的自动分析规则，则停留在 `pending`，用户可手动对单个告警或批量告警点击“分析”。
- 分析过程使用主智能体的**告警分析场景**，并行调用预置工具和子智能体。

---

### 六、知识入库机制

仅当告警状态变为 `confirmed` 时，才根据分析结果生成知识条目并写入知识库。

**触发条件**：
- 用户点击“确认”按钮。
- 可选的“确认并入库”操作会触发知识生成。

**生成规则**：
- 从 `analysis_result` 中提取摘要、根因、建议，结合原始事件生成一条标准化的故障处理条目。
- 自动添加标签：来源、告警名称、主机、服务。
- 调用知识库 API 写入，并返回 `knowledge_entry_id` 关联到告警。

**用户可修改**：确认前，可编辑标题、补充细节，确保知识质量。

**未确认不入库**：任何 `dismissed` 或未经确认的告警均不会生成知识条目。

---

### 七、权限控制

- `alerts.view`：查看告警列表和详情。
- `alerts.analyze`：手动触发分析。
- `alerts.confirm`：确认告警（同时控制入库能力）。
- `alerts.dismiss`：忽略告警。
- `alerts.assign`：指派处理人。

管理员可将这些权限分配给不同角色的用户，确保操作规范。

---

### 八、与已有模块的无缝衔接

- **数据接入**（运维中心 → 数据接入）产生的事件自动沉淀为告警。
- **场景运维**的“告警分析”场景可被手动或自动调用，处理告警。
- **自动化**的触发规则驱动批量自动分析。
- **知识库**（AI中心 → 知识库）接收确认后的告警知识，丰富 RAG 来源。
- **对话**提供深度交互分析的入口。

告警中心作为运维事件的集散地，让平台从数据到知识形成完整闭环，贯彻“1分钟发现、5分钟定位”的效率要求，并通过人工确认保证知识库质量。
