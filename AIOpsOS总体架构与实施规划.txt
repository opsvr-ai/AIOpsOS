## AIOpsOS 总体架构与实施规划

本文档整合此前所有设计内容，从顶层视角描述 **AIOpsOS（AI运维智能操作系统）** 的架构、功能、通信、组件、交互等全维度规划，作为落地开发的统一蓝图。

---

### 1. 产品定位与目标

**AIOpsOS** 是面向企业运维的**多智能体操作系统**，通过云-边-端协同实现从事件感知、故障诊断、预案执行到知识沉淀的完整闭环，且达成 **1分钟发现、5分钟定位、10分钟恢复** 的运维1-5-10标准。

**核心能力**：
- 多智能体协作（主智能体拆解任务、调度子智能体、调用技能/MCP工具）
- 告警自动分析、根因定界、日常巡检、开关门巡检等预制场景
- 知识持续沉淀（LLM-Wiki编译式知识库，阻断人工确认后才入库）
- 安全自动化执行（客户端沙箱、工作流确认）
- 开放可扩展（插件化通知渠道、动态注册 Skill 与 MCP 服务）

---

### 2. 总体架构：管控分离与云边协同

平台分为三大平面：

```
┌─────────────────────────────────────────────────────────┐
│                    前端展示层 (React)                     │
│  - 运维中心 / AI中心 / 控制中心                           │
└────────────┬─────────────────────────────┬──────────────┘
             │ HTTPS/WSS                   │
┌────────────▼───────────┐  ┌──────────────▼──────────────┐
│     管控平面 (Control)   │  │    数据平面 (Execution)      │
│  - 用户/权限/审计        │  │  - 多智能体引擎 (LangGraph)     │
│  - 场景/工具/智能体 CRUD │  │  - 对话/会话/流式交互         │
│  - 定时任务/触发器配置   │  │  - 告警分析/根因定界          │
│  - 知识库管理            │  │  - 工具调用 (Skill/MCP)       │
│  - 客户端Profile管理     │  │  - 消息通知/渠道回调          │
│  - 系统设置 (品牌/LDAP)  │  │  - Kafka 事件消费            │
└────────────┬───────────┘  └──────────────┬──────────────┘
             │                             │
             │ 共享 PostgreSQL + Redis + Kafka
             │
┌────────────▼─────────────────────────────┐
│          客户端 Agent (边缘执行)           │
│  - 安全通讯与注册                          │
│  - Skills 沙箱执行 (Docker/Wasm)          │
│  - 数据采集 (metrics/logs)                │
│  - 本地自治 (规则引擎)                     │
│  - 自更新                                 │
└──────────────────────────────────────────┘
```

**特点**：
- **管控与执行分离**：管控平面专注配置管理，执行平面专注实时交互与分析运行。两者可独立扩展和部署，也支持单体模式快速启动。
- **云端智能结合边缘执行**：服务端负责重推理、规划，客户端 Agent 负责安全执行与数据采集，实现 “思考在云，动作在边”。

### 3. 功能模块全景

| 模块 | 主要功能 | 归属平面 |
|------|---------|----------|
| **运维中心** | 对话交互、告警中心、场景运维（告警分析/根因定界/巡检）、自动化（定时任务+触发规则）、数据接入、历史 | 数据平面（执行） |
| **AI中心** | 智能体定义（主/子）、工具注册（Skill/MCP）、知识库（LLM-Wiki）、模型配置、记忆进化 | 结合管控与执行 |
| **控制中心** | 消息渠道管理、身份与权限（RBAC+LDAP）、系统管理（品牌/安全设置）、客户端Agent管理 | 管控平面 |
| **客户端Agent** | 服务器/虚拟机/网络/存储的智能代理，执行Skills、采集、自治 | 边缘 |

### 4. 通信架构

```
┌───────────┐      HTTPS/WSS       ┌───────────────┐
│  Web 前端  │ ◄──────────────────► │  Nginx 网关    │
└───────────┘                       └───┬───────┬───┘
                                        │       │
                     /api/control/*      │       │ /api/execution/*, /ws
                                        ▼       ▼
                                 ┌─────────┐ ┌─────────┐
                                 │ Control │ │Execution│
                                 └────┬────┘ └──┬──┬───┘
                                      │         │  │
                                      │  Kafka   │  │ WebSocket (mTLS)
                                      ▼         ▼  ▼
                               ┌─────────┐  ┌──────────┐
                               │  Worker  │  │ Client   │
                               │  Beat    │  │ Agent    │
                               └─────────┘  └──────────┘
```

- **前端 ↔ 后端**：HTTPS REST / WebSocket。通过 Nginx 根据路径前缀分流管控与执行。
- **服务端内部**：共享 PostgreSQL、Redis、Kafka。Celery Worker 通过 Redis 消息队列执行定时或异步任务。
- **数据接入**：外部系统（Prometheus、Zabbix等）通过 Kafka 推送事件或 MCP 拉取数据。
- **客户端 Agent ↔ 服务端**：WebSocket over TLS 双向认证（mTLS），推送指令、配置，回传执行结果与采集数据。

### 5. 组件规划

#### 5.1 后端核心组件

| 组件 | 技术选型 | 职责 |
|------|----------|------|
| API 框架 | FastAPI | 提供 REST 接口，自动文档 |
| 智能体引擎 | LangGraph (Python) | 构建主智能体有状态图，支持子智能体调度 |
| 工具管理 | 自研 ToolManager | 统一管理 Skill 和 MCP 工具，热加载、缓存、限流 |
| MCP 集成 | mcp SDK + 自研连接池 | 管理多个 MCP 服务器连接及工具发现 |
| 技能执行器 | Docker 沙箱 + HTTP 代理 | 安全执行用户 Python 函数或调用外部 API |
| 记忆系统 | MemoryCore (pgvector) | 长期记忆存储与检索，支持自动注入和进化 |
| 知识库 | LLM-Wiki 编译引擎 | raw/ → wiki/ 编译，交叉引用，定期 lint |
| 消息通知 | 插件化渠道 (基类 + 注册) | 钉钉、企微、微信、Webhook 等，支持用户自定义 |
| 定时调度 | Celery + RedBeat | 动态 Cron 任务，周期性执行场景 |
| 事件处理 | Kafka Consumer | 告警事件标准化、去重、匹配触发规则 |
| 认证授权 | JWT + RBAC + LDAP | 用户认证与细粒度权限控制 |

#### 5.2 前端主要组件

| 组件 | 说明 |
|------|------|
| ChatContainer | 对话主框架，管理消息流 |
| ChatMessageList | 虚拟滚动消息列表，支持多种卡片类型 |
| IntentCard | 意图识别卡片（场景标签、实体提取） |
| PlanCard | 任务规划卡片，展示子任务及状态 |
| ExecutionCard | 工具/子智能体调用卡片（折叠/展开详情） |
| KnowledgeRefCard | 知识引用卡片 |
| SecurityConfirmCard | 高风险操作确认卡片 |
| InlineParameterForm | 动态参数收集表单 |
| ChatInput | 输入区（含命令面板、附件上传、场景快捷标签） |
| ScenarioSelector | 场景选择器 |
| CronBuilder | Cron 表达式可视化构建器 |
| TriggerConditionBuilder | 触发条件可视化构建器 |
| AgentProfileManager | 客户端配置模板管理 |

#### 5.3 客户端 Agent 核心模块

| 模块 | 说明 |
|------|------|
| Transport | mTLS WebSocket 客户端，负责安全通讯 |
| ConfigCache | 加密存储服务端下发的 Profile，验签并热加载 |
| SkillExecutor | 沙箱内执行技能脚本 (Docker/Wasm) |
| Collector | 系统指标/日志采集与本地降噪 |
| RuleEngine | 本地自治规则评估与动作执行 |
| SelfUpdater | 接收升级指令并安全更新自身 |

### 6. 交互与体验规划

**设计原则**：极简科技风（深色主题、大量留白、微动效）、所见最少但所得最多。

**主要交互流程**：
- **对话驱动运维**：用户在输入框输入问题或通过 `/` 命令选择场景，智能体返回意图、规划、执行全过程卡片，最终给出结论与引用。
- **自动响应**：告警事件到达后，自动创建分析会话，前端无操作时仅显示通知，点开即可查看分析详情。
- **人机协作**：高风险操作弹出安全确认卡片；信息缺失时展示参数收集表单，用户提交后继续。
- **知识闭环**：告警确认后，弹出知识编辑窗口，修改后一键编译入库，并在 wiki 可查。
- **个性化助理**：每个用户可配置专属工具集、偏好场景、通知渠道，智能体在规划时自动生效。

### 8. 部署规划

**快速体验**（单体模式）：
```bash
docker-compose -f deploy/docker-compose.allinone.yml up -d
```
单个服务实例包含管控+执行，适合开发与演示。

**生产部署**（管控分离）：
```bash
docker-compose -f deploy/docker-compose.yml up -d
```
独立启动 control、execution、worker、beat、kafka-consumer、frontend 容器。

**客户端部署**：使用 Ansible Playbook 批量安装 Agent，支持 Linux/Windows。

**扩展性**：执行平面与 Worker 可水平扩展，Kafka 分区支持高并发事件处理，数据库读写分离（按需）。

---

### 9. 开发阶段建议

1. **MVP (4-6周)**：实现告警分析核心链路（事件→分析→确认→入库），基础对话，基础告警列表，单渠道通知，单体部署。
2. **扩展 (6-8周)**：自动化（定时/触发），多场景（巡检/定界），人工介入完善，知识库 LLM-Wiki 运转，客户端 Agent 基础执行，多消息渠道。
3. **成熟 (8-12周)**：智能体进化/记忆，性能优化（缓存/并行），触发规则可视化，客户端 Profile 管理，权限精细化，自修复能力增强。
4. **企业化 (按需)**：高可用多数据中心，对接 ITSM/CMDB，合规审计增强，多租户，离线自治策略，Kubernetes 部署。

---

### 10. 保障措施

- **安全**：四层沙箱执行、mTLS 全链路、JWT细粒度权限、API限流、操作全程审计。
- **稳定**：断线重连、客户端离线缓存、心跳超时告警、Lane Queue（会话隔离）。
- **可追溯**：会话全量记录、智能体推理链路存储、客户端操作审计日志、知识编译变更日志。
